# 要件定義書（ドラフト v0.2）
## 旅行計画・記録アプリ（仮称：tabiru）

> v0.1からの主な変更：プラットフォームを **ネイティブアプリ** に決定。技術方針・アーキテクチャ、機能要件の詳細化、非機能要件を追加。
> 一行サマリ：**「Funlidayの計画・記録機能」＋「Googleワンタップで気軽に共同編集」＋「Funlidayにない動画・インスタ用データの自動生成」**

---

## 1. 確定した前提

| 項目 | 決定 |
|---|---|
| ターゲット | 学生・若者中心の複数人グループ旅行 |
| 認証 | Google認証必須（Sign in with Google） |
| 参加モデル | 招待URL → Googleワンタップ → グループ参加 |
| プラットフォーム | **ネイティブアプリ（iOS / Android）** |
| スコープ外 | 割り勘・旅費管理 |

---

## 2. 技術アーキテクチャ（推奨）

Next.js単体ではネイティブアプリを作れないため、役割を分ける構成を推奨する。

| レイヤー | 技術（推奨） | 役割 |
|---|---|---|
| ネイティブアプリ本体 | **Expo / React Native** | 計画・記録・動画生成・カメラ・位置情報・共有 |
| Web招待入口 | **Next.js** | 招待URLの受け口、未インストール者の閲覧、アプリへのディープリンク |
| バックエンド／API | Next.js API もしくは BaaS | 認証・データ・メディアの仲介 |
| 認証 | Google OAuth | ログイン・本人性 |
| データ／リアルタイム同期 | Firebase / Supabase 等 | 共同編集・写真合流 |
| メディア保存 | クラウドストレージ＋CDN | 写真・生成動画の保管と配信 |
| 地図 | 地図API（要選定） | スポット検索・ルート・表示 |
| 動画レンダリング | 端末 or サーバー（要選定） | 旅程ダイジェスト動画の生成 |

代替案：Next.js＋Capacitor（Webアプリをネイティブの殻で包む）。コード再利用に優れるが、WebView方式のため動画・大量写真など重い処理で体感が落ちる。

## 4. 非機能要件

### NFR-1 認証・認可
- Google OAuthを用いる。トークンは安全に保管（端末セキュアストレージ）。
- すべての操作はロール（オーナー／編集者／閲覧者）に基づき認可される。
- 退会・グループ離脱時のアクセス失効を即時反映する。

### NFR-2 データ同期・整合性
- 複数メンバーの同時編集に対応するリアルタイム同期。
- 競合解決方針を定義する（例：項目単位の last-write-wins、もしくは並べ替えに強い手法）。**要決定。**
- 同期遅延の目標：オンライン時、他メンバーの変更が概ね数秒以内に反映。

### NFR-3 オフライン対応
- 旅行先は電波が不安定な前提。**オフラインファースト**で設計する。
- オフライン時も「旅程の閲覧」「写真・メモの記録」は可能とし、再接続時に自動同期。
- 同期前の編集はキューに保持し、競合時の扱いを定義する。

### NFR-4 メディア処理・ストレージ
- 写真は端末側で圧縮・サムネイル生成してから送信し、転送量と表示速度を最適化。
- 原本／表示用の解像度を分けて保持することを検討。
- ストレージ容量・転送量はコストに直結するため、保存ポリシー（保持期間、画質、上限）を定義。**要決定。**

### NFR-5 動画生成・レンダリング
- 生成方式：**端末側レンダリング**（サーバーコスト低・写真が端末外に出ない）か **サーバー側**（品質・速度が安定）かを選定。**要決定。**
- 体感目標：標準的な旅程（数十枚規模）で過度に待たせない（目標値は要調整）。
- BGMは権利処理済み素材のみ使用。人気楽曲の埋め込みは行わない。

### NFR-6 ディープリンク／導線
- 招待URLは Universal Links / App Links でアプリを起動。未インストール時はストア誘導後に当該グループへ復帰。
- 未インストール者向けにWeb（Next.js）での閲覧フォールバックを用意。

### NFR-7 セキュリティ・プライバシー
- 写真は割り勘金額より機微度が高い前提で扱う。
- 通信はTLS、保存データは暗号化。アクセスは認可で厳格に制御。
- ユーザーによるデータ削除（写真・旅行・アカウント）を提供。
- 位置情報・カメラ等の権限取得は目的を明示し最小限に。

### NFR-8 パフォーマンス
- アプリ起動・写真グリッドのスクロール・タイムライン表示が滑らかであること（目標値は要調整）。
- 大量写真時もメモリ・描画が破綻しない（遅延読み込み等）。

### NFR-9 スケーラビリティ・コスト
- 地図API・メディアストレージ・動画レンダリングが主要なランニングコスト。利用増に対して費用が線形に膨らまない設計を意識。
- 無料配布前提のため、コスト上限とマネタイズの接続点（保存・高画質書き出し等）を後続で定義。

### NFR-10 端末・OS対応
- iOS／Androidの対応下限バージョンを定義。**要決定。**

## 5. 未決の技術論点（次に決める）

| # | 論点 | 補足 |
|---|---|---|
| T1 | バックエンド基盤の選定 | Firebase（リアルタイム・認証・ストレージが揃う）／ Supabase（Postgresベース）／ 自前。同期・認証・メディアをまとめて担う中核 |
| T2 | 動画レンダリングの場所 | 端末 or サーバー（NFR-5） |
| T3 | 地図サービスの選定 | Google Maps Platform／その他。検索・ルート・表示のコスト（NFR-9） |
| T4 | 同期の競合解決方式 | last-write-wins ／ 並べ替えに強い手法（NFR-2） |
| T5 | メディア保存ポリシー | 保持期間・画質・容量上限（NFR-4） |
| T6 | 対応OS下限 | iOS／Androidの最低バージョン（NFR-10） |
